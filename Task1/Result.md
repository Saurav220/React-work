const STATUS_CLASSES = {
pending: "status-badge--pending",

This is the idiomatic React way to handle conditional class mapping. You avoided a switch statement or a chain of if/else blocks, which is exactly what I'd expect from someone thinking about maintainability. When we add a 6th status next quarter (and we will), it's a one-line change. Excellent. ✅

🟡 Suggestions — Things I'd Push Back On In a Real PR

1. No PropTypes — this would block merge on our codebase.

Right now, if someone uses <StatusBadge /> without passing status, or passes status={42}, there's no development-time warning. We use PropTypes across all our components (until we migrate to TypeScript next quarter). Add them:

jsx

import PropTypes from "prop-types";
StatusBadge.propTypes = {
status: PropTypes.string,
};
StatusBadge.defaultProps = {
status: "unknown",
};
💡 Mentor note: With defaultProps, you can also remove the status || "unknown" fallback in the JSX — the default prop handles it. Keeps the render function cleaner. Note that defaultProps is deprecated in React 18.3+ for function components — the modern pattern is to use default parameter destructuring:

jsx

function StatusBadge({ status = "unknown" }) { ... }
This is what we'll eventually standardize on.

2. Consider co-locating the label with the class map.

Right now, the class name lives in STATUS_CLASSES, but the display label is dynamically generated by formatStatus. This works today, but what if a product manager says "Change 'On Hold' to 'Paused'"? You'd need to add a special case to your format function. A more scalable pattern:

jsx

const STATUS_CONFIG = {
pending: { className: "status-badge--pending", label: "Pending" },
"in-progress": { className: "status-badge--in-progress", label: "In Progress" },
completed: { className: "status-badge--completed", label: "Completed" },
failed: { className: "status-badge--failed", label: "Failed" },
"on-hold": { className: "status-badge--on-hold", label: "On Hold" },
};
This makes the component a single source of truth for both styling and labeling. formatStatus is clever, but clever isn't always better than explicit in production code.

🧠 At NovaCraft, we prefer explicit over clever — because at 2 AM during an incident, nobody wants to debug a string transformation.

3. Accessibility — missing semantic cues.

A screen reader will read "Completed" as plain text, which is fine. But adding a subtle ARIA attribute makes it even better for assistive technology:

jsx

<span
className={`status-badge ${badgeClass}`}
role="status"
aria-label={`Status: ${label}`}

> This isn't blocking, but it's the kind of thing that earns you trust with the QA team.

4. Demo page inline styles — acceptable but noted.

The inline styles in App.js are fine for a demo, but I want to flag it: in our codebase, even demo/storybook pages use CSS files. Just something to keep in mind for future tasks.
